# Step 4.4: PCAP ストレージ - トラブルシューティング

## 問題: PCAP ストレージが2回連続で未完了

### 原因分析

**根本原因: サブエージェントのコンテキスト消失**

前回のセッションで Phase 4 の3つのストレージバックエンド (SQLite, JSONL, PCAP) を並列サブエージェントで実装した。

1. SQLite エージェント → ✅ 正常完了、ファイル書き込み成功
2. JSONL エージェント → ✅ 正常完了、ファイル書き込み成功
3. **PCAP エージェント → ❌ コンテキスト切り替え(会話がコンテキスト上限に到達)でエージェントが消失**

エージェントの完了を待機している間にメインコンテキストが上限に達し、セッションが打ち切られた。その結果:
- PCAPエージェントの作業結果は失われた
- `lib.rs` はスケルトン (`// netcap-storage-pcap` の1行) のまま残った
- `converter.rs` は前のセッションで作成されていたが、テスト不十分な状態だった

### 技術的問題

1. **pcap-file 2.0.0 の `PcapWriter` に `get_mut()` メソッドがない**
   - `BufWriter` でラップすると flush できない
   - 解決: `std::fs::File` を直接使用 (unbuffered write)

2. **pcap-file のパッケージバージョン不一致**
   - 初回のエージェント起動時、gcc-12 の deb パッケージ URL が変更されていた (`12.2.0-14` → `12.2.0-14+deb12u1`)
   - これによりビルド環境の構築に時間がかかり、エージェントのタイムアウトに寄与

3. **Cargo.toml の `bytes` と `http` 依存追加が必要だった**
   - converter.rs のテストで `bytes::Bytes` を使用
   - Cargo.toml に既に追加されていたが、テストでのimport漏れ

### 解決策

手動でPCAPストレージを実装:
- `converter.rs`: Ethernet + IPv4 + TCP ヘッダ付き擬似パケット構築 (6テスト)
- `lib.rs`: PcapStorage + StorageBackend 実装 (7テスト)
- 全13テスト通過

### 再発防止

- サブエージェントの並列実行時は、コンテキスト使用量を監視
- 重要なエージェントは foreground で実行し、完了を確認してから次に進む
- コンテナリスタート対策として、ビルド環境のセットアップスクリプトを永続化
